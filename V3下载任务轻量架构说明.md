# V3 下载任务轻量架构说明

## 1. 目标

`v3` 的核心目标不是兼容旧任务模型，而是围绕 HLS 大量分片下载场景，做一套更轻、更稳、更容易扩展的任务运行时。

这版重点解决的是：

- 去掉高频 `task_item` 状态写库
- 降低 SQLite 写放大
- 降低大任务下的内存和 I/O 压力
- 保留任务级管理能力
- 保持现有 `m3u8` 解析与重写链路

## 2. 设计原则

- `tasks` 表只保存任务级快照，不保存分片级实时状态
- 静态分片清单入库，但不承担高频状态更新
- 高频变化优先留在内存 runtime
- 磁盘只保存轻量 `progress.json`
- aria2 回调只驱动内存状态，不做逐分片落库
- 对账只做补偿，不做主流程

## 3. 核心数据模型

### 3.1 `tasks`

任务主表，保存任务级信息：

- `id`
- `name`
- `original_url`
- `total_segments`
- `downloaded_segments`
- `total_items`
- `done_items`
- `failed_items`
- `output_dir`
- `created_time`
- `updated_time`
- `finished_time`
- `status`
- `proxied_content`

它的职责只有一个：给前端和管理接口提供任务级快照。

### 3.2 `task_manifest`

静态清单表，只存分片元数据，不存下载状态：

- `task_id`
- `seq`
- `filename`
- `url`
- `item_type`

特点：

- 任务创建时批量写入一次
- 删除任务时批量删除
- 下载过程中不更新

这个表替代了 `manifest.json`，好处是：

- 降低 runtime 对大对象文件的依赖
- 需要查具体分片时可以直接走 SQL
- 恢复 runtime 时不需要解析大 JSON

### 3.3 `progress.json`

每个任务一个轻量进度文件，路径在任务缓存目录下：

- `tid`
- `f`
- `seg_done`
- `done`
- `u`

对应含义：

- `tid`: 任务 ID
- `f`: 失败文件名列表
- `seg_done`: 已完成分片数
- `done`: 已完成文件数
- `u`: 更新时间

这版 `progress.json` 不再保存全量 `pending` 列表，也不再保存失败错误字符串。

## 4. runtime 结构

内存中的运行态在 [internal/task/manager.go](D:\go\src\com.fy.test\hls-accelerator\internal\task\manager.go) 的 `taskRuntime` 中，当前主要包括：

- `taskID`
- `totalItems`
- `totalSegments`
- `segmentBySeq`
- `remaining map[string]uint32`
- `failed map[string]struct{}`
- `dispatching map[string]struct{}`
- `gidToFile map[string]string`
- `fileToGID map[string]string`
- `remainingSegments`
- `paused`
- `dirty`
- `dirtySince`
- `lastAccessAt`

关键点：

- runtime 不再常驻整份 manifest，更不会常驻所有 URL
- `remaining` 不是存整份 `ManifestItem`，而是 `filename -> seq`
- `segmentBySeq` 只保存“这个 seq 是否算分片”的轻量布尔信息
- `failed` 只存失败文件集合，不再常驻错误字符串
- 分发时真正需要的 `url/type`，按批次从 `task_manifest` 回查
- 活跃下载和绑定关系只保留在内存中

这比旧版 `task_item` 模型更轻，也比“runtime 常驻整份 manifest”更省内存。

## 5. 任务创建流程

入口在代理层 `POST /api/v1/tasks`。

流程如下：

1. 校验 URL 和任务名
2. 解析 `m3u8`
3. 构建下载项列表
4. 构建 `TaskManifest`
5. 写入 `tasks`
6. 批量写入 `task_manifest`
7. 初始化 `progress.json`
8. 加载 runtime
9. 启动分发协程

这条链路里：

- `m3u8` 的解析与重写逻辑保持现状
- 新架构只改任务编排和状态管理

## 6. 下载分发模型

每个任务启动一个 dispatch goroutine。

分发逻辑：

1. 从 runtime 中 `claimPending(batchSize)`
2. 按文件名批量回查 `task_manifest`
3. 按批次构造 aria2 `addUri` 请求
4. 用 `BatchAddURIs` 批量提交
5. 将返回的 `gid` 绑定到文件名
6. 任务暂停时，对刚绑定的 `gid` 补 pause

分发批次默认是固定大小，不再依赖数据库逐条扫描分片状态。

## 7. 完成/失败处理

aria2 事件通过通知通道进入 `progressNotificationWorker()`。

当前策略：

- `onDownloadComplete`
  - 只更新 runtime
  - 移除 `remaining`
  - 从 `failed` 中清理
  - 标记 `dirty`

- `onDownloadError`
  - 只更新 runtime
  - 写入失败文件集合
  - 标记 `dirty`

不再在这里做逐分片数据库写入。

## 8. 刷盘策略

### 8.1 为什么不实时刷盘

如果每个完成/失败事件都立刻刷：

- 会频繁重写 `progress.json`
- 会频繁更新 `tasks`
- 吞吐会重新被 I/O 拖住

所以 `v3` 用的是“内存先变，后台按节奏刷盘”。

### 8.2 `dirty` 机制

当 runtime 状态发生变化时：

- 设置 `dirty = true`
- 首次变脏时记录 `dirtySince`

后台 `flushDirtyLoop()` 按状态自适应判断是否需要落盘。

### 8.3 刷盘窗口

当前实现是：

- `downloading`: 约 `2s`
- `paused`: 约 `400ms`
- `completed/failed`: 约 `150ms`

这样做的效果是：

- 下载高峰期减少小写入
- 暂停/失败/完成等管理动作更快收敛

### 8.4 刷盘内容

`flushRuntime()` 做两件事：

1. 写 `progress.json`
2. 更新 `tasks` 表快照

任务完成或彻底失败后，会立即触发 runtime 卸载。

## 9. aria2 清理策略

### 9.1 基本原则

在 `v3` 中，aria2 只承担“下载执行器”职责，不承担长期状态存储职责。

长期真实状态来源仍然是：

- `tasks`
- `task_manifest`
- `progress.json`

因此，aria2 里的历史任务结果需要主动清理，避免长期运行时结果集无限增长。

### 9.2 为什么不能依赖 aria2 自己保留历史

原因有三点：

- aria2 内存结果集会随着时间增长而膨胀
- `tellStatus / tellStopped` 的扫描成本会越来越高
- aria2 重启后，历史队列本身也不是可靠持久化来源

所以 `v3` 的设计目标是：

- 活跃任务依赖 aria2
- 已结束任务不依赖 aria2 历史结果

### 9.3 删除任务时按目录立即清理

当前实现里，不再对终态任务做统一的 5 分钟延迟清理。

只在任务进入 `deleted` 后，于异步删除流程里直接清理 aria2。

这样做的原因是：

- 逻辑更简单，少一套额外的内存状态和轮询后台任务
- `deleted` 语义本身就是“任务已经不再需要保留”
- 日常兜底仍然有每天凌晨 `03:00` 的全局 purge

### 9.4 删除清理是按任务目录做的

`v3` 不会为每个已结束任务持久化历史 `gid` 列表。

原因是：

- `gid` 是 aria2 会话级临时标识
- 任务长期主键是 `task_id` 和 `filename`
- 如果把历史 `gid` 重新持久化，又会引入额外状态负担

所以清理时走的是“按任务目录反查”策略：

1. 用 `task_id` 算出任务缓存目录
2. 向 aria2 查询该目录下的：
   - `active`
   - `waiting`
   - `stopped`
3. 汇总得到该目录关联的 `gid`
4. 批量执行：
   - `forceRemove`
   - `removeDownloadResult`

对应实现：

- [internal/downloader/aria2.go](D:\go\src\com.fy.test\hls-accelerator\internal\downloader\aria2.go) 的
  - `TellActive`
  - `TellWaiting`
  - `TellStopped`
  - `CleanupTaskByDir`

### 9.5 删除任务时的处理

删除任务时会有两层处理：

1. 任务先切到 `deleted`
2. 立刻从任务列表和 runtime 中移除
3. 后台异步删除：
   - active/waiting/stopped 关联下载
   - 本地缓存目录
   - `progress.json`
   - `tasks / task_manifest`

### 9.6 每天凌晨 3 点的全局 purge

除了删除任务时的即时清理，系统还会做一层全局低频兜底：

- 每天凌晨 `03:00`
- 执行一次 `aria2.purgeDownloadResult()`

作用：

- 清理零散历史结果
- 防止某些异常路径下未按任务清干净的残留继续堆积

这个全局 purge 不是主清理手段，而是兜底策略。

### 9.7 aria2 启动参数建议

为了防止 aria2 自身历史结果无限增长，建议启动时加上：

```bash
--max-download-result=2000
```

它的作用是限制 aria2 内部保留的历史结果数量。

这和业务层清理策略是互补关系：

- `max-download-result` 控制上限
- `deleted` 任务的异步删除负责按目录精细清理
- 每日 `purgeDownloadResult()` 负责全局兜底

## 10. 对账模型

对账是补偿机制，不是主状态来源。

对账入口：

- 周期循环
- 手工调用 `/api/v1/tasks/sync`

逻辑：

1. 只处理 `downloading / paused / failed`
2. 遍历 runtime 里未完成项
3. 检查文件是否已落盘且 `.aria2` 不存在
4. 再结合 aria2 队列状态补偿
5. 统一刷 runtime 快照

这保证了即使 aria2 事件漏掉，任务也能最终收敛。

## 11. runtime 生命周期

为了避免任务越跑越多导致内存持续上涨，`v3` 增加了 runtime 淘汰策略。

当前规则：

- `downloading`: 常驻
- `completed / failed`: 刷盘成功后立即卸载
- `paused`: 空闲超过 `10` 分钟后卸载

再次访问时，会从：

- `tasks`
- `task_manifest`
- `progress.json`

重新恢复 runtime。

## 12. 前端接口

当前前端页面只依赖任务级接口：

- `GET /api/v1/tasks`
- `POST /api/v1/tasks`
- `POST /api/v1/tasks/{id}/pause`
- `POST /api/v1/tasks/{id}/resume`
- `POST /api/v1/tasks/{id}/retry`
- `POST /api/v1/tasks/sync`
- `DELETE /api/v1/tasks/{id}`

页面不再依赖任务详情或 `task_item` 级别接口。

## 13. 相比旧方案的主要变化

### 13.1 去掉的东西

- `task_item` 状态表
- 分片级高频数据库写入
- 分片级前端详情接口
- `manifest.json`

### 13.2 保留的东西

- `m3u8` 解析与重写逻辑
- aria2 作为下载执行器
- SQLite 作为本地任务元数据存储
- 本地缓存目录结构

### 13.3 新增的东西

- `task_manifest` 静态清单表
- 轻量 `progress.json`
- 内存 runtime
- runtime 淘汰机制
- 自适应刷盘节奏

## 14. 当前实现的收益

- SQLite 写压力显著下降
- 下载高峰期不再逐分片落库
- `progress.json` 体积更小
- 活跃 runtime 内存占用更低
- 静态清单可 SQL 查询
- runtime 不再常驻全量 URL 数据
- aria2 历史结果可持续清理
- 前后端任务管理逻辑更简单

## 15. 当前实现的边界

- 分片级错误详情不再长期保存
- 失败项目前只保留文件名集合
- 对账仍然有文件系统扫描成本
- SQLite 依然不是高并发数据库，只是负担比旧版轻很多
- aria2 清理仍然依赖目录归属正确、缓存目录结构稳定
- `completed / failed` 的历史结果主要依赖每日 purge 和 `max-download-result` 控制

## 16. 后续可继续优化的方向

- 对账由逐文件 `os.Stat` 改成单次 `ReadDir` 建索引
- 给失败文件增加可选的少量错误样本，而不是全量错误字符串
- 为任务列表增加更细的观测指标，例如 runtime 数量、脏任务数量、刷盘耗时
- 为 aria2 清理增加更细的执行指标，例如待清理任务数、最近清理耗时、清理命中条数
